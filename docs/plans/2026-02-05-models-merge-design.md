# Models 目录合并设计

## 概述

将 `lib/models/` 和 `lib/models_new/` 合并为单一的 `lib/models/` 目录，以 `models_new` 的结构为基础。

## 背景

- **`lib/models/`**：84个文件，旧版模型，包含通用枚举、类型定义和本地存储模型
- **`lib/models_new/`**：466个文件，新版模型，按API端点组织，更全面的API响应模型
- 两目录存在重复和交叉引用
- 代码库中 `models_new` 的引用（462处）远多于 `models`

## 核心原则

1. **以 `models_new` 为主体**：models_new 有466个文件，已覆盖大部分API，且被462处代码引用
2. **优先保留新版本**：遇到重复模型时，保留 models_new 的版本
3. **逐步验证更新**：不使用全局替换，而是逐个文件验证导入更新
4. **保持结构一致**：在 models_new 中创建 common/ 目录存放通用类型

## 目标目录结构

```
lib/models/
├── common/              # 从 models/common/ 迁移的通用枚举和类型
│   ├── enums.dart
│   └── types.dart
├── article/             # 从 models_new 迁移（API 响应模型）
├── video/
├── live/
├── user/
├── search/
└── ...                  # 其他功能模块
```

## 执行步骤

### 阶段 1：准备工作

1. **创建目录结构**：在 `models_new` 中创建 `common/` 目录
2. **备份分支**：创建专门的分支进行合并（如 `merge-models`）
3. **分析依赖**：识别所有需要更新导入的文件（约462个）

### 阶段 2：迁移 models 独有内容

1. **迁移 common 目录**：将 `models/common/` 中的枚举和通用类型迁移到 `models_new/common/`
2. **识别 models 独有模型**：对比两个目录，找出只在 `models/` 中存在的模型
3. **迁移独有模型**：将这些模型复制到 `models_new/` 的相应位置
4. **处理 Hive 注解**：检查 `.g.dart` 文件，确保 Hive 本地存储模型正确迁移

### 阶段 3：重命名目录

1. **临时重命名**：将当前的 `models/` 重命名为 `models_old/`
2. **重命名 models_new**：将 `models_new/` 重命名为 `models/`
3. **验证结构**：确保新的 `models/` 目录结构完整

### 阶段 4：更新导入语句（逐步验证）

1. **获取文件列表**：使用 grep 找到所有包含 `models_new` 导入的文件
2. **批量处理**：每次处理 10-20 个文件，逐个更新：
   - 将 `package:piliplus/models_new/...` 改为 `package:piliplus/models/...`
   - 运行 `fvm flutter analyze` 检查错误
   - 修复任何导入问题
3. **测试验证**：每处理一批文件后运行测试

### 阶段 5：清理和验证

1. **删除 models_old**：确认无误后删除旧目录
2. **全量测试**：运行 `fvm flutter test` 确保所有测试通过
3. **静态分析**：运行 `fvm flutter analyze` 确保无警告

## 风险控制

### 高风险点

1. **导入路径错误**：462个文件的导入更新可能遗漏或出错
2. **类型冲突**：某些模型可能在不同上下文中使用不同版本
3. **Hive 存储兼容性**：本地存储的模型迁移可能导致数据读取失败
4. **循环依赖**：合并后可能出现新的循环引用问题

### 回滚策略

**检查点设置**：
- 每个阶段完成后创建 git commit（便于回滚）
- 关键阶段前打 tag（如 `before-models-merge`）

**回滚方案**：
1. **部分回滚**：如果某阶段失败，回滚到上一个 commit
2. **完全回滚**：如果合并失败，恢复到合并前的分支状态
3. **数据保护**：Hive 本地数据库可能需要清理和重建

### 验证清单

**每个文件更新后验证**：
- [ ] 导入路径正确
- [ ] 无静态分析错误
- [ ] 相关功能正常工作

**完成后验证**：
- [ ] 所有测试通过
- [ ] `fvm flutter analyze` 无错误
- [ ] 应用能正常启动
- [ ] 主要功能（视频播放、登录、搜索等）正常
- [ ] Hive 数据读写正常

## 时间估算

- 阶段 1（准备）：约 30 分钟
- 阶段 2（迁移）：约 2-3 小时（需仔细对比）
- 阶段 3（重命名）：约 10 分钟
- 阶段 4（更新导入）：约 3-4 小时（逐个验证）
- 阶段 5（清理验证）：约 1-2 小时

**总计**：约 7-11 小时
